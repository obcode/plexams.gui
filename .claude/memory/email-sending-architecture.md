E-Mail-Versand lebt in `src/lib/email/`: `emails.js` (per-type `EmailDef` defs + `emailGroups` registry, grouped by workflow phase) and `EmailSender.svelte` (self-contained card for ONE send type). Overview page `/email` (`+page.svelte`) renders all `emailGroups`. Nav has a top-level "E-Mails" menu → `/email`.

Each send is a graphql-ws subscription `<key>(run: Boolean!) { level text }` ending on a DONE LogLine — same transport/LogLine type as validation/generation. `EmailSender` reuses `getConvert`/`getWsClient` from `$lib/validation/wsClient` (shared lazy socket + ansi-to-html) and renders the stream as a terminal (PROGRESS lines in-place). See [[validation-architecture]].

UI contract: default button = Probelauf (`run:false`, mails only the planner); separate "Wirklich senden …" button reveals an inline confirm before `run:true`. The backend rejects sends while a validation/transfer/other email runs (an ERROR line "a validation or another transfer/email is running, cannot start now" + DONE) — `EmailSender` detects that text and shows a yellow Hinweis (warning), NOT a hard error. Cf. [[validation-write-lock]].

`EmailSender` is deliberately standalone so it can be embedded on existing pages later (user's stated plan). It now also takes optional `extraArgs` ({name:{type,value}}) — builds the subscription args dynamically beyond `run` — and a `disabled` prop. Send-subscriptions NOT yet offered (backend later): primuss-data, nta-\*. E4 `sendEmailPublishedInvigilations(run)` is wired (bulk-only): on the /email hub (Veröffentlichung group) and on the attachments page Aufsichtskalender section, gated on ≥1 uploaded invigilation-image (page tracks both managers' attachments via their `change` event). Missing PNGs → WARN line, stream continues; dry-run mails the test address. Wired send keys: sendEmailConstraints, sendEmailExaHM, sendEmailPrepared, sendEmailDraft, sendEmailPublishedExams, sendEmailPublishedRooms, sendEmailInvigilations, sendEmailInvigilationsMissing, plus E3 cover-pages: `sendEmailCoverPages(run)` (bulk, in emails.js "Deckblätter" group → /email, and on the attachments page) and `sendEmailCoverPage(teacherID:Int!, run)` (single, attachments page with an examiner picker using `extraArgs`). Cover-page sends find PDFs from the uploaded cover-page attachments or backend `coverPages.dir`; a missing PDF yields a per-examiner ERROR line while the stream continues. NB: subscriptions are NOT in the committed schema.graphql (no Subscription type there at all) and EmailSender uses a plain template string, so there are no editor GraphQL diagnostics for them.

**Attachment store** (separate from sending): `src/lib/email/attachments.js` (helpers) + `AttachmentManager.svelte` (reusable per-`kind` card) + page `/email/attachments` (nav "E-Mails" → "Anhänge"). `kind` ∈ 'cover-page' (key=teacher ID) | 'invigilation-image' (key=invigilator ID). Binary UPLOAD is REST direct to the backend base (`backendBase()` = PUBLIC_PLEXAMS_SERVER minus `/query`), NOT GraphQL: `POST {base}/upload/email-attachment` (single file: kind,key,file) and `POST {base}/upload/email-attachments-zip` (zip: kind,file — server derives key from last digit-run in each filename, returns `{stored, skipped[]}`). HTTP 409 on upload = validation/transfer/mail running → treat as Hinweis (`{blocked:true}`), not error. List/clear ARE GraphQL via /api proxies (`/api/email/emailAttachments?kind=`, `/api/email/clearEmailAttachments`): `emailAttachments(kind){key filename size uploadedAt}` / `clearEmailAttachments(kind)`. The attachments page server-load reconciles both kinds via AttachmentManager `expectedKeys` (shows missing): cover-pages against ALL examiners in the plan (`examerInPlan` {mainExamer,mainExamerID}), invigilation-images against all invigilators (`invigilatorTodos.invigilators[].teacher`); labels are teacher shortnames joined from `teachers(fromZPA:false)`. Page section order: Aufsichtskalender (invigilation-image) on top, Deckblätter (cover-page) below. invigilation-image PNGs are browser-generated on `/zpa/invigilator_requirements` (card capture via html-to-image `toBlob`, key = `invigilator.teacher.id` read from the card's `data-invig-name` attr). That page has a "Bilder auf Server hochladen" button (`uploadAllPng`) that captures each visible card and uploads it directly via `uploadAttachment()` (respects the search filter; stops on a 409 blocked). The existing "Kalender als PNG (ZIP)" export names files `<id>.png`, so the invigilation-image AttachmentManager also has `acceptZip` as a fallback (key derived from filename digits).
